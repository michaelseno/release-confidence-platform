# Bug Report

## 1. Summary
During HITL validation of Enhanced `rcp config init` Default Profile System, `rcp audit run` successfully invoked the orchestrator Lambda but the Lambda failed during module import:

```text
[ERROR] Runtime.ImportModuleError:
Unable to import module 'apps.backend.handlers.orchestrator_handler':
No module named 'requests'
```

The failure is a backend Serverless/Lambda packaging defect: application code imports `requests`, but the Serverless deployment artifact does not include Python third-party dependencies.

## 2. Investigation Context
- Source of report: HITL validation / CloudWatch runtime log.
- Active branch: `feature/profile_driven_config_init`.
- Related workflow: `rcp config init` -> `rcp audit validate` -> `rcp audit create` -> `rcp audit run`.
- Current reported state:
  - `rcp config init` succeeded.
  - `rcp audit validate` succeeded.
  - `rcp audit create` succeeded after prior fixes.
  - `rcp audit run` reached Lambda invocation.
- Failing runtime component: `release-confidence-platform-dev-coreEngineOrchestrator`, handler `apps.backend.handlers.orchestrator_handler.handler`.
- Local artifact inspection command run safely without deploying:

```bash
cd infra
serverless package --stage dev
```

This only generated local files under ignored `infra/.serverless/`; it did not deploy or mutate cloud resources.

## 3. Observed Symptoms
- Failing workflow: manual audit run after successful config and audit creation.
- Exact CloudWatch error:

```text
[ERROR] Runtime.ImportModuleError:
Unable to import module 'apps.backend.handlers.orchestrator_handler':
No module named 'requests'
```

- Expected behavior: Lambda imports `apps.backend.handlers.orchestrator_handler`, constructs the orchestrator, loads S3/DynamoDB-backed audit configuration, executes endpoint checks, writes raw results to S3, and updates audit metadata in DynamoDB.
- Actual behavior: Lambda fails at import/init before orchestrator business logic runs because `requests` is missing from the deployed Python path.
- Why earlier CLI steps passed: `config init`, `audit validate`, and `audit create` execute in the operator environment and use installed local project dependencies. They do not verify the deployed Lambda artifact contains backend runtime dependencies. `audit run` uses async Lambda invocation, so the invoke API can accept the event while the handler later fails in Lambda runtime initialization.

## 4. Evidence Collected
Files/config inspected:
- `pyproject.toml`
- `infra/serverless.yml`
- `infra/package.json`
- `infra/.serverless/serverless-state.json` generated by local package
- `infra/.serverless/release-confidence-platform.zip` generated by local package
- `apps/backend/handlers/orchestrator_handler.py`
- `apps/backend/orchestrator/service.py`
- `apps/backend/runner/api_runner.py`
- `src/release_confidence_platform/storage/lambda_client.py`
- `src/release_confidence_platform/core/manual_run_service.py`

Dependency definition findings:
- No backend-specific requirements file exists under `apps/backend/` or `infra/`. Repository search found only `apps/mock-target-api/requirements.txt`.
- Root `pyproject.toml` declares the dependency:

```toml
dependencies = [
  "boto3>=1.34,<2",
  "requests>=2.31,<3",
]
```

- The backend code imports `requests` in `apps/backend/runner/api_runner.py` line 9 and uses `requests.Session()` in `ApiRunner.__init__()`.
- Import path to the failing handler is configured in `infra/serverless.yml` line 89: `apps.backend.handlers.orchestrator_handler.handler`.

Serverless packaging findings:
- `infra/serverless.yml` has only a local custom plugin:

```yaml
plugins:
  - ./plugins/stage-guard
```

- There is no `serverless-python-requirements` plugin, no `custom.pythonRequirements`, no Lambda layer, and no dependency install target configured.
- `infra/package.json` devDependencies include only `serverless`; no Python packaging plugin is installed.
- `package.patterns` includes repository Python source paths (`../apps/backend/**`, `../packages/**`) but does not include site-packages, a requirements install directory, or a layer.
- `package.individually` is not configured; the generated state shows a single service artifact shared by functions.
- Generated `serverless-state.json` confirms runtime `python3.11`, handler `apps.backend.handlers.orchestrator_handler.handler`, empty per-function package configs, no layers, and artifact path `infra/.serverless/release-confidence-platform.zip`.

Artifact evidence:
- Generated artifact: `infra/.serverless/release-confidence-platform.zip`.
- Artifact metadata from local inspection:

```text

requests/__init__.py: False
urllib3/__init__.py: False
certifi/__init__.py: False

```

- The artifact contains application modules such as `apps/backend/handlers/orchestrator_handler.py` and `apps/backend/runner/api_runner.py`, but it does not contain `requests` or its transitive dependencies.
- CloudWatch's error progressed far enough to import `apps.backend.handlers.orchestrator_handler`; therefore the monorepo source path is present enough for handler discovery. The missing module is specifically the third-party dependency `requests`.

Runtime compatibility findings:
- `infra/serverless.yml` provider runtime is `python3.11`.
- Root `pyproject.toml` requires `>=3.11,<3.12`.
- `requests` is pure Python; no Lambda architecture/wheel mismatch is indicated by the evidence. Missing packaging, not binary incompatibility, is the root issue.

Deployment command/path findings:
- Serverless service root is `infra/`; `infra/package.json` scripts run `serverless package/deploy --stage ...` from that directory.
- If `serverless-python-requirements` is added without `fileName`, it will look for requirements relative to the service directory, so the chosen requirements file path must match the `infra/` working directory assumption.

## 5. Execution Path / Failure Trace
1. Operator runs `rcp audit run` after audit config and metadata creation.
2. `ManualRunInvocationService.run()` calls `LambdaInvocationClient.invoke()` with the configured orchestrator function.
3. The Lambda invoke API accepts the async event, so CLI-side validation can appear successful/generic.
4. AWS Lambda initializes handler `apps.backend.handlers.orchestrator_handler.handler`.
5. `orchestrator_handler.py` imports `CoreEngineOrchestrator` from `apps.backend.orchestrator.service`.
6. `service.py` imports `ApiRunner` from `apps.backend.runner.api_runner`.
7. `api_runner.py` executes `import requests` at module import time.
8. The deployed zip lacks `requests`, causing Lambda runtime import failure before handler code runs.

## 6. Failure Classification
- Primary category: Environment / Configuration Issue.
- More specific subtype: Deployment packaging/dependency bundling defect.
- Severity: Blocker.

Severity justification: This blocks HITL validation and production use of `audit run`; the orchestrator cannot start in Lambda, so no audit execution, raw S3 result, or DynamoDB terminal metadata update can occur.

## 7. Root Cause Analysis
Confidence label: Confirmed Root Cause.

Immediate failure point:
- Lambda runtime import fails at `apps.backend.runner.api_runner import requests` because `requests` is not present in the deployed artifact/PYTHONPATH.

Underlying root cause:
- Backend deployment does not install/package Python dependencies. `requests>=2.31,<3` exists in root `pyproject.toml`, but Serverless packaging is source-only and is not configured to consume that dependency declaration or a backend requirements file.

Supporting evidence:
- `requests` dependency exists only in root `pyproject.toml`, not in a backend/infra requirements file used by Serverless.
- `infra/serverless.yml` lacks `serverless-python-requirements`, `pythonRequirements`, dependency layers, or any package include for installed site-packages.
- `infra/package.json` lacks `serverless-python-requirements`.
- Generated Lambda zip contains backend source files but has zero `requests` entries and zero `requests-*.dist-info` entries.
- CloudWatch reports `No module named 'requests'`, matching the artifact contents exactly.

Plausible contributing factors:
- The backend has no dedicated dependency manifest colocated with the Serverless service. The root Python project dependency list is sufficient for local CLI/test use but is currently disconnected from Lambda packaging.
- Runtime diagnostics are too generic for async Lambda runtime failures; operator CLI can report invocation acceptance while the real failure is only visible in CloudWatch.

## 8. Confidence Level
High.

The CloudWatch error, import path, dependency declarations, Serverless config, and generated zip contents all point to the same missing dependency packaging issue. No evidence indicates orchestrator business logic, CLI payload construction, audit lifecycle state, Python version, or architecture incompatibility as the primary failure.

## 9. Recommended Fix
Likely owner: backend/infrastructure.

Recommended developer fix:
1. Add an explicit Lambda dependency manifest for the backend, for example `apps/backend/requirements.txt` containing at least:

   ```text
   requests>=2.31,<3
   ```

   Consider whether to include/pin `boto3>=1.34,<2` as well. Lambda includes boto3, but vendoring it avoids runtime-version drift if the code relies on newer SDK behavior.

2. Wire Serverless packaging to that manifest from the `infra/` service directory:
   - add `serverless-python-requirements` to `infra/package.json` devDependencies;
   - add it to `infra/serverless.yml` plugins after/before the local stage guard as appropriate;
   - configure `custom.pythonRequirements.fileName` to the selected manifest path relative to `infra/`, e.g. `../apps/backend/requirements.txt`;
   - set packaging options intentionally, for example `slim: true`; consider `dockerizePip: non-linux` or `true` for future native dependencies even though `requests` is pure Python.

3. Keep monorepo source packaging patterns for `apps/backend/**` and `packages/**`, but add/verify excludes so local virtualenvs, caches, and Node dependencies are not accidentally bundled.

4. Add a packaging regression check, preferably in infra/unit validation, that runs `serverless package --stage dev` and asserts the artifact contains:
   - `apps/backend/handlers/orchestrator_handler.py`
   - `apps/backend/runner/api_runner.py`
   - `requests/__init__.py`
   - `urllib3`, `certifi`, `charset_normalizer`, and `idna` entries or the equivalent installed dependency set.

Runtime diagnostics improvement recommendation:
- Enhance CLI/operator diagnostics for Lambda runtime failures separately from invoke API failures:
  - Preserve existing invoke API mappings such as function not found and permission denied.
  - When feasible, support a validation/diagnostic mode using synchronous `InvocationType=RequestResponse` for HITL/debug runs, parse `FunctionError` and sanitized payload/log tail, and map common failures to operator-friendly codes:
    - `LAMBDA_DEPENDENCY_IMPORT_ERROR` for `Runtime.ImportModuleError` / `No module named ...`.
    - `CONFIG_BUCKET_NOT_FOUND` for S3 `NoSuchBucket`/404 config bucket failures.
    - `AUDIT_METADATA_TABLE_NOT_FOUND` for DynamoDB `ResourceNotFoundException` on metadata table operations.
    - `AWS_CREDENTIALS_NOT_FOUND` / existing credentials-focused code for missing local credentials.
  - Avoid exposing raw payload secrets; sanitize function names, AWS request ids, and error messages.

## 10. Suggested Validation Steps
After the packaging fix:
1. Local package validation, no deploy:
   - Run from `infra/`: `serverless package --stage dev`.
   - Inspect `infra/.serverless/release-confidence-platform.zip` and verify `requests/__init__.py` and transitive dependencies are present.
   - Verify `apps/backend/handlers/orchestrator_handler.py` and `packages/**` are still present.
2. Redeploy validation, required for HITL completion:
   - Deploy the fixed backend Serverless artifact to the same dev stage/region/account used by the active HITL run.
   - Confirm the deployed Lambda code package is updated (new version/hash/timestamp).
3. Runtime validation:
   - Run `rcp audit run` with the same client/audit context or a fresh run id.
   - Confirm CloudWatch no longer shows `Runtime.ImportModuleError` / `No module named 'requests'`.
   - Confirm orchestrator logs progress to `run_started` / `run_completed` or a domain-specific sanitized failure rather than import failure.
4. Data-plane verification:
   - S3: verify raw results were written under the expected raw results key for `client_id/audit_id/run_id`.
   - DynamoDB: verify metadata row transitions to a terminal status (`completed` or a meaningful domain failure), with `raw_result_s3_key` populated on success.
5. Regression checks:
   - Re-run `rcp audit validate` and `rcp audit create` for a fresh validation audit to ensure packaging changes did not affect CLI/local flows.
   - Add/execute the artifact-content regression check in CI or infra validation.

## 11. Open Questions / Missing Evidence
- Which exact deploy command/path was used for the currently deployed Lambda (`infra` scripts vs another workflow)? Repository config indicates `infra/` is the intended Serverless service root, but the live deployment command history was not provided.
- Whether the deployed live artifact exactly matches the locally generated artifact cannot be proven without downloading the Lambda package or deployment artifact from AWS. The local packaging config reproduces the missing dependency state and matches the CloudWatch error.
- Whether boto3 should also be vendored depends on project runtime policy; it is not implicated in this failure.

## 12. Final Investigator Decision
Ready for developer fix.

The root cause is confirmed as missing Python dependency bundling in the backend Serverless Lambda artifact. A live redeploy is required after the fix before HITL can validate `audit run` end-to-end.
