# Bug Report

## 1. Summary

During HITL validation on `feature/profile_driven_config_init`, `rcp audit create` failed with a generic `UNEXPECTED_ERROR` when run against config files generated by the enhanced `rcp config init` workflow.

Primary issue: the generated default starter bundle is validated only in internal `template_mode=True`, but downstream operator commands (`audit validate` / `audit create`) use execution validation that rejects the default empty `endpoints` array. A secondary error-ordering defect in `audit create` constructs AWS clients before config validation, so local AWS/profile failures can mask the actionable config validation error as `UNEXPECTED_ERROR`.

## 2. Investigation Context

- Source of report: HITL manual validation after QA approval.
- Branch context: `feature/profile_driven_config_init` must remain the active correction branch.
- Related feature/workflow: Enhanced `rcp config init` Default Profile System -> generated local config bundle -> `rcp audit create`.
- User action: non-dry-run `rcp audit create` using generated `.local-configs/.../client_config.json`, `audit_config.json`, and `endpoints.json` with `--stage dev`.

## 3. Observed Symptoms

Observed output:

```text
ERROR: audit create failed
stage: dev
code: UNEXPECTED_ERROR
message: Unexpected operator CLI failure
next_step: correct the error and retry
```

Expected behavior:

- Generated config-init starter configs should be validation-safe per feature requirements.
- If `audit create` cannot proceed because generated files are incomplete starter templates, the CLI should return a structured, actionable validation/configuration error, not `UNEXPECTED_ERROR`.
- `audit create` should not initialize AWS clients before local config validation, because that masks local config defects and makes validation depend on AWS profile availability.

Local reproduction notes:

- The exact user `.local-configs/...` files were not present in the repository checkout.
- I generated a representative bundle with matching deterministic IDs under `/var/folders/7y/zdp6qp9n4dz00dn9f5c3n9lr0000gn/T/opencode/rcp_bug_investigation/...`.
- Direct validation of the generated default bundle with normal execution validation failed with:

```text
ConfigError CONFIG_VALIDATION_ERROR Endpoint config must include at least one endpoint
```

- The same bundle passed only with `template_mode=True`.
- Running the CLI module directly in this local shell was blocked by missing `boto3` (`ModuleNotFoundError`) before CLI dispatch. This is an environment limitation of the investigation shell, not proof that the user's environment lacked `boto3`.

## 5. Execution Path / Failure Trace

Likely path for the reported command:

1. `rcp audit create` dispatches to `services.create_command()`.
2. `create_command()` loads stage config, then constructs `AwsClientFactory` before validating the local JSON files.
3. If AWS profile/session/client construction fails in the user's environment, that non-`EngineError` bubbles to `main.main()` and is rendered as `UNEXPECTED_ERROR`.
4. If AWS setup succeeds, `AuditCreationService.create_from_files()` validates the generated configs in runtime mode.
5. Runtime validation calls `validate_endpoint_config(..., allow_empty=False)` and rejects the generated default `endpoints: []` with `CONFIG_VALIDATION_ERROR`.

The immediate observed failure is the generic error wrapper. The underlying workflow incompatibility is that generated starter configs are accepted by config-init template validation but are not accepted by the downstream validation/create workflow operators are instructed to run.

## 6. Failure Classification

- Primary classification: Contract Mismatch
- Contributing classification: Application Bug / error handling and validation ordering
- Severity: Blocker

Severity justification: this occurred during HITL after QA approval and blocks completion of the advertised config-init -> validate/create onboarding workflow. The CLI also hides the actionable failure behind `UNEXPECTED_ERROR`.

## 7. Root Cause Analysis

Confidence label: Most Likely Root Cause

Immediate failure point:

- A non-`EngineError` exception in `audit create` is caught by `main.main()` and rendered as `UNEXPECTED_ERROR`.

Underlying root cause:

- `audit create` initializes AWS infrastructure before local config validation, so AWS/session/profile errors can mask local generated-config issues.
- Generated default config-init bundles contain empty endpoints by design, but `audit validate` / `audit create` use runtime validation that requires at least one endpoint. This contradicts the feature requirement that generated files validate safely without AWS/network access.

Contributing factors:

- `config init` uses an internal-only `template_mode=True` validation path, but operator-facing `audit validate` has no equivalent starter/template mode.
- `main.main()` intentionally sanitizes unexpected exceptions, but no known AWS setup exceptions are translated into structured `EngineError` values.

## 8. Confidence Level

Medium-high.

The generated-config/runtime-validation mismatch is directly reproduced and supported by code. The exact `UNEXPECTED_ERROR` cause is strongly supported by the code path that constructs AWS before validation and wraps all non-`EngineError` exceptions, but the user's machine-specific exception type was not available in the report.

## 9. Recommended Fix

Likely owner: `dev-backend`.

Fix guidance:

1. In `src/release_confidence_platform/operator_cli/services.py`, reorder `create_command()` so local config validation happens before constructing `AwsClientFactory` for non-dry-run execution. Do not allow AWS profile/session/client setup to mask local config errors.
2. Align the config-init starter contract with downstream commands:
   - Either add an operator-facing starter/template validation mode for `rcp audit validate` and make config-init next steps point to that mode before real endpoints are added, or
   - Change config-init defaults/next steps so `audit create` is not recommended until endpoints are populated, or
   - If product truly requires create-safe starter configs, generate safe sample endpoints by default or otherwise ensure runtime validation passes without weakening production/destructive safeguards.
3. Translate known AWS setup exceptions from `AwsClientFactory` / boto3 session/client creation into structured `StorageError` or `ConfigError` messages such as missing AWS profile/stage configuration, rather than generic `UNEXPECTED_ERROR`.
4. Add regression coverage for config-init generated outputs flowing into `audit validate` and `audit create --dry-run` without AWS construction, including the omitted `--include-sample-endpoints` default.

Caution: Do not make `audit create` persist audits with empty executable endpoint sets unless product explicitly approves that behavior. Empty endpoints may be valid for starter templates but unsafe/confusing for real audit creation.

## 10. Suggested Validation Steps

- Generate a default dev bundle with `rcp config init --client-name "Layer 1 Validation Client"`.
- Run the documented validation next step and confirm it exits with the intended structured result.
- Run `rcp audit create --dry-run` against the generated bundle and confirm no AWS clients are constructed before local validation.
- Run non-dry `rcp audit create` in an environment without the configured AWS profile and confirm the CLI returns a structured AWS/profile configuration error if local config validation has already passed.
- Run the same flow with `--include-sample-endpoints` and verify runtime validation/create behavior is explicit and documented.

## 11. Open Questions / Missing Evidence

- The exact user-side Python exception behind `UNEXPECTED_ERROR` was not captured. A traceback or debug log would confirm whether it was `ProfileNotFound`, another boto3/botocore setup exception, or a later storage serialization issue.
- Need product decision on whether generated default empty-endpoint bundles are intended to be only template-validation-safe or fully `audit create`-safe.
- Need confirmation of the exact config-init next-step text the user saw, because the current inspected `NEXT_STEPS` list does not include a literal `rcp audit create` command.

## 12. Final Investigator Decision

Ready for developer fix.

The workflow blocker and code paths are sufficiently identified. Additional user traceback would improve certainty for the exact generic-error trigger but is not required to fix the validation-ordering and generated-config contract mismatch.
